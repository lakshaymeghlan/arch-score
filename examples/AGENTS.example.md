# AGENTS.md

Instructions for AI coding agents working in this repository. Follow these system-design conventions.

This is a **cli** project (primary language: TypeScript). Follow the conventions below when reading, writing, or refactoring code.

**Architecture:** Target a **layered** structure. Keep IO at the edges: a thin bin/ entry parses args and delegates to commands/, which call pure core/ logic. This keeps the business logic testable without spawning the CLI.

**Folder structure:** Place new code according to this layout:

```
bin/                 # executable entry, arg parsing only
src/
  commands/          # one module per command, thin
  core/              # pure business logic, no process/IO
  adapters/          # fs, network, process boundaries
  config/
tests/
```

**Apply these standards (the project currently falls short on them):**
- Grow the suite around the most critical and most-changed modules.
- Keep IO at the edges: a thin bin/ entry parses args and delegates to commands/, which call pure core/ logic. This keeps the business logic testable without spawning the CLI.
- Add a CI workflow that runs the test suite on every push/PR.
- Add a committed .env.example listing every required variable (with safe placeholder values).
- Introduce a single error-handling boundary (middleware or top-level handler) that maps errors to responses/exit codes.
- Document the high-level architecture and key decisions (consider arch-score --emit-md).
- No `commands/` directory — recommended for cli projects.
- Handle, log, or rethrow in catch blocks; never swallow errors silently.
- Add CONTRIBUTING.md describing setup, conventions, and the PR process.

**Per-category guidance:**
- **Testing Architecture**: Grow the suite around the most critical and most-changed modules. Add a CI workflow that runs the test suite on every push/PR.

**General system-design rules:**
- Keep entry points thin; put business logic in dedicated modules.
- Externalize configuration via environment variables; never hardcode secrets or endpoints.
- Handle errors explicitly — never swallow them in empty catch blocks.
- Add or update tests alongside code changes; cover core logic and IO boundaries.
- Use structured logging; add timeouts/retries around network calls in services.
- Avoid circular dependencies and god-modules; depend on abstractions, not concretes.
